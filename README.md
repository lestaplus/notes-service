## Документація по розробленому веб-застосунку

### Призначення застосунку
**mywebapp (Notes Service)** - це простий REST API сервіс для збереження, перегляду та керування текстовими нотатками. Він дозволяє створювати нові записи та переглядати існуючі через веб-інтерфейс (HTML) або у форматі JSON.

### Налаштування середовища (локально)
Для розробки та запуску застосунку на локальній машині необхідно:
1. Встановити **Node.js** (версія 20.x).
2. Встановити **PostgreSQL**.
3. Створити базу даних `notes_db` та користувача `student` з паролем `mypassword`.
4. Скопіювати файл `config.example.json` у новий файл `config.json` та за потреби змінити налаштування підключення до бази даних.
5. Встановити залежності, виконавши команду `npm install` у корені проєкту.

### Як запустити застосунок
1. Виконайте міграцію бази даних (створення таблиць): `node migrate.js`
2. Запустіть застосунок: `node app.js`

Для швидкого розгортання застосунку в ізольованому середовищі з використанням Docker:

3. Виконайте команду: `docker compose up -d --build`

Для зупинки контейнерів виконайте: `docker compose down`

## Документація по API-ендпоінтах

Застосунок підтримує Content Negotiation. Заголовок `Accept: text/html` повертає HTML-сторінку, `Accept: application/json` — дані у форматі JSON.

* **`GET /health/alive`** — перевірка життєздатності сервісу (завжди повертає `HTTP 200 OK`).
* **`GET /health/ready`** — перевірка готовності (повертає `200 OK`, якщо є зв'язок з БД, інакше `500`).
* **`GET /`** — кореневий ендпоінт, повертає HTML-сторінку зі списком усіх доступних маршрутів.
* **`GET /notes`** — вивести список усіх нотаток (id, title).
* **`POST /notes`** — створити нову нотатку. Очікує JSON: `{"title": "Назва", "content": "Текст"}`.
* **`GET /notes/<id>`** — вивести повний вміст нотатки за її ID (id, title, created_at, content).

## Документація по розгортанню

### Базовий образ віртуальної машини
Для розгортання використовується офіційний ISO-образ **Ubuntu Server 24.04 LTS**, завантажений з офіційного сайту Ubuntu.

### Вимоги до ресурсів віртуальної машини
* **CPU:** 1 ядро
* **RAM:** 1024 MB (1 GB)
* **Disk:** 10 GB
* **Мережа:** NAT із налаштованим прокиданням портів (Port Forwarding): 
  * Хост `2222` -> Гість `22` (для SSH)
  * Хост `8080` -> Гість `80` (для HTTP)

### Спеціальні налаштування при встановленні ОС
1. Стандартна розбивка диску (LVM).
2. Обов'язково відмітити встановлення **OpenSSH Server** для віддаленого доступу.

### Як увійти на ВМ
* **Спосіб:** SSH через прокинутий порт.
* **Команда:** `ssh -p 2222 admin@127.0.0.1`
* **Credentials:** логін `admin` (або інший, створений при встановленні ОС) та ваш пароль.

### Як завантажити та запустити автоматизацію
Усі кроки з розгортання автоматизовано скриптом `setup.sh`.

1. Скопіюйте проєкт на ВМ: `scp -P 2222 -r ./* admin@127.0.0.1:~/app_setup/`
2. Зайдіть на ВМ через SSH та перейдіть у папку: `cd ~/app_setup/`
3. Надайте права на виконання: `chmod +x setup.sh`
4. Запустіть скрипт: `sudo ./setup.sh`

*(Примітка: Після виконання скрипта початковий користувач блокується для безпеки. Наступні входи слід виконувати з-під створеного користувача `student`).*

## CI/CD Конвеєр (Лабораторна робота №3)

У проєкті налаштовано автоматизований CI/CD конвеєр за допомогою **GitHub Actions**:
1. **Continuous Integration (CI):** При пуші в гілку `main` або створенні Pull Request код автоматично перевіряється лінтером (ESLint) та покривається юніт-тестами (Jest). Заблоковано злиття PR, якщо перевірки не пройдено.
2. **Container Registry:** Успішно протестований код збирається в Docker-образ та автоматично публікується у GitHub Container Registry (GHCR). Застосовується автоматичне тегування (`latest`, унікальний `sha`, та `stable` для релізів).
3. **Continuous Deployment (CD):** Власний Self-hosted Runner автоматично підключається до цільової ноди, завантажує свіжий образ з GHCR, виконує розгортання та автоматичну HTTP-верифікацію застосунку.

### Запуск готового образу з GHCR

Ви можете завантажити вже зібраний production-образ з репозиторію командою:
```bash
docker pull ghcr.io/lestaplus/notes-service:latest
```

### Приклад швидкого локального запуску

Щоб не створювати конфігураційні файли вручну, ви можете просто скопіювати і виконати ці команди. Вони автоматично створять ізольовану мережу, тестовий конфіг і запустять контейнери без конфліктів із локальними сервісами:

**Для користувачів Linux / Mac / WSL (Bash):**
```bash
# 1. Створюємо віртуальну мережу
docker network create notes-net

# 2. Створюємо тестовий конфіг
echo '{"port": 3000, "database": {"host": "notes-db", "port": 5432, "user": "postgres", "password": "mypassword", "database": "postgres"}}' > config.json

# 3. Запускаємо базу даних у мережі
docker run -d --name notes-db --network notes-net -e POSTGRES_PASSWORD=mypassword postgres:15

# 4. Запускаємо застосунок
docker run -d --name notes-app --network notes-net -p 3000:3000 -v $(pwd)/config.json:/etc/mywebapp/config.json ghcr.io/lestaplus/notes-service:latest
```

*Примітка: Якщо під час виконання команд на Linux виникає помилка `permission denied`, виконайте їх з правами суперкористувача (додавши `sudo` перед `docker`), або додайте вашого користувача до групи `docker` (`sudo usermod -aG docker $USER`).*

**Для користувачів Windows (PowerShell):**
```powershell
# 1. Створюємо віртуальну мережу
docker network create notes-net

# 2. Створюємо конфіг (з UTF-8 кодуванням)
[System.IO.File]::WriteAllText("$PWD\config.json", '{"port": 3000, "database": {"host": "notes-db", "port": 5432, "user": "postgres", "password": "mypassword", "database": "postgres"}}')

# 3. Запускаємо базу даних
docker run -d --name notes-db --network notes-net -e POSTGRES_PASSWORD=mypassword postgres:15

# 4. Запускаємо застосунок
docker run -d --name notes-app --network notes-net -p 3000:3000 --mount type=bind,source="$PWD\config.json",target=/etc/mywebapp/config.json ghcr.io/lestaplus/notes-service:latest
```

Після виконання команд застосунок буде доступний за адресою: `http://localhost:3000`

## Лабораторна робота №4: Декларативне керування інфраструктурою (Terraform & Ansible)

### Варіант індивідуального завдання (Варіант 27)
* **Застосунок:** Notes Service (Сервіс нотаток)
* **База даних:** PostgreSQL
* **Порт застосунку:** 3000

### Архітектура розподіленої системи
Замість розгортання всіх компонентів на одному вузлі, систему розподілено на дві окремі віртуальні машини в межах однієї Host-Only мережі VirtualBox:
1.  **VM1 (`worker_node`)** - розгортається Nginx (Reverse Proxy) та Node.js веб-застосунок.
2.  **VM2 (`db_node`)** - розгортається СУБД PostgreSQL.

---

### Інструкція з розгортання інфраструктури

### Етап 1: Provisioning (Terraform)
Створення та запуск віртуальних машин повністю автоматизовано за допомогою Terraform. Через специфіку роботи провайдера `terra-farm/virtualbox` на ОС Windows, замість базових Cloud-образів використано оптимізований образ `generic/ubuntu2204`, який гарантує стабільне виділення мережевих інтерфейсів та DHCP-адрес.

1. Перейдіть до папки з конфігурацією Terraform:
```bash
cd iac/terraform
```

2. Виконайте ініціалізацію провайдера:
```bash
terraform init
```

3. Створіть віртуальні машини:
```bash
terraform apply
# (Введіть yes для підтвердження)
```

*Примітка щодо налаштування мережі: Перед початком переконайтеся, що у VirtualBox увімкнено DHCP-сервер для Host-Only адаптера (`VirtualBox Host-Only Ethernet Adapter`). Якщо після створення машин інтерфейс NAT (`eth1`) перебуває у стані DOWN, його необхідно активувати на кожній ВМ командою `sudo dhclient eth1` для забезпечення доступу до інтернету.*

### Етап 2: Configuration Management (Ansible)
Конфігурація операційної системи, встановлення пакетів, керування користувачами та деплой застосунку реалізовано декларативно за допомогою Ansible-ролей.

1. Перейдіть до папки з конфігурацією Ansible (рекомендується запускати з середовища WSL2):
```bash
cd iac/ansible
```

2. Відкрийте файл `inventory.ini` та вкажіть актуальні IP-адреси, які отримали створені віртуальні машини:
```ini
[workers]
worker_node ansible_host=192.168.56.105 ansible_user=vagrant

[db]
db_node ansible_host=192.168.56.106 ansible_user=vagrant
```

3. Перед першим запуском переконайтеся, що ваш SSH-ключ скопійовано на віртуальні машини:
```bash
ssh-copy-id vagrant@192.168.56.105
ssh-copy-id vagrant@192.168.56.106
```

4. Запустіть плейбук для повної автоматичної конфігурації обох нод:
```bash
ansible-playbook -i inventory.ini playbook.yml
```

### Перевірка ідемпотентності
При повторному запуску команди `ansible-playbook -i inventory.ini playbook.yml` система перевіряє поточний стан конфігурації. Усі таски (окрім специфічних випадків, таких як синхронізація коду) повертають статус `ok` замість `changed`, що підтверджує повну ідемпотентність розгортання.

## Інструкція з тестування системи
Після успішного виконання Ansible-плейбука розподілена система готова до роботи:

1. Головна сторінка: `http://192.168.56.105/` - повертає стартову сторінку через Nginx reverse proxy.
2. Маршрут бізнес-логіки: `http://192.168.56.105/notes` - відображає таблицю нотаток (автоматичний запуск міграцій виконано сервісом systemd перед стартом застосунку).
3. Health Check зв'язку: `http://192.168.56.105/health/ready` - повертає `OK`, що підтверджує успішне підключення веб-застосунку з `worker_node` до PostgreSQL на `db_node`. Прямий доступ до БД ззовні Host-Only мережі суворо обмежений конфігурацією `pg_hba.conf`.

## Особливості реалізації та Troubleshooting (ОС Windows)

Під час виконання роботи на хостовій операційній системі Windows із використанням провайдера `terra-farm/virtualbox` було виявлено та успішно вирішено кілька критичних архітектурних проблем:

1. **Проблема з ковзанням IP-адрес (VirtualBox Guest Additions):**
   При використанні офіційних мінімальних Cloud-образів Ubuntu плагін Terraform на Windows не міг коректно зчитувати виділені DHCP-адреси через специфіку взаємодії з гостьовою ОС, що призводило до таймауту (`Wait VM until ready`). 
   * **Рішення:** Базовий образ було замінено на стабільний та оптимізований під VirtualBox образ `generic/ubuntu2204`. Оскільки цей образ не підтримує первинне конфігурування через `cloud-init` на етапі provisioning, для першого підключення Ansible було використано стандартного дефолтного користувача `vagrant` із доступом за заздалегідь згенерованим SSH-ключем.

2. **Керування користувачами (Заміна cloud-init на Ansible):**
   Оскільки створення користувача `ansible` з адміністративними правами через `cloud-init` було неможливим через обмеження гостьового образу, цю логіку було повністю перенесено на етап конфігурації.
   * **Рішення:** Роль `base` в Ansible повністю бере на себе створення всіх передбачених завданням користувачів (`teacher`, `student`) та файлу `gradebook`, забезпечуючи повністю ідемпотентний та декларативний підхід до налаштування ОС.

3. **Сплячі мережеві інтерфейси в базовому образі:**
   Після підняття ВМ інтерфейс NAT (`eth1`), який відповідає за вихід в інтернет, за замовчуванням перебував у стані DOWN.
   * **Рішення:** Мережеві карти були примусово активовані за допомогою DHCP-клієнта (`sudo dhclient eth1`) безпосередньо перед запуском конфігураційного плейбука, що дозволило Ansible безперешкодно завантажити та налаштувати пакети Nginx, Node.js та PostgreSQL з офіційних репозиторіїв.
